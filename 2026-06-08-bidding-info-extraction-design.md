# 招标资料关键信息提取与废标风险提示系统设计

## 1. 项目定位

本系统是一个本地部署的网页系统，用于处理信息化软件、系统集成、服务类采购项目的招标公告和招标文件。

系统运行在本地电脑或 NAS 上，用户通过浏览器访问。系统接收网页公告、PDF、DOCX、扫描件和附件包，自动解析并提取项目登记本信息、关键时间、金额、招标人、代理机构，以及招标文件中可能导致废标或否决投标的关键条款。

系统不判断企业是否能投，不做企业资质、业绩、人员能力与招标要求的匹配，也不输出"建议投标/建议放弃"结论。系统只提取文件中的客观信息和风险条款，供人工复核。

## 2. 一期 MVP 范围

一期包含以下能力：

1. 本地网页系统。
2. Docker 方式部署。
3. 项目新建和资料上传。
4. 招标公告网页内容录入或抓取。
5. PDF、DOCX、扫描件、附件包解析。
6. 项目登记本字段提取。
7. 废标、否决投标、无效投标条款提取。
8. `★`、`*`、`▲` 等特殊标记条款专项识别。
9. 信息化软件、系统集成、服务类采购重点条款提取。
10. 人工复核、修改和确认。
11. 项目列表、项目详情、字段复核、风险清单页面。
12. Excel 导出项目登记本和风险清单。

一期不包含：

1. 企业资质库和自动匹配。
2. 是否建议投标的决策判断。
3. 自动生成投标文件。
4. 报价测算。
5. CRM、合同管理、回款管理。
6. 复杂多租户 SaaS 权限。
7. 钉钉、企业微信、邮件集成。
8. 补遗文件自动差异对比。

## 3. 部署方式

系统采用 Docker 部署，目标是同一套服务既能在本地电脑调试，也能部署到 NAS 内网环境。

推荐容器组成：

1. Web 前端服务：提供浏览器界面。
2. 后端 API 服务：处理项目、文件、解析、抽取、复核、导出。
3. 文档解析服务：处理 PDF、DOCX、OCR、网页正文解析。可以与后端合并为一个容器，也可以后续拆分。
4. 数据库：一期推荐 SQLite，文件挂载到宿主机目录。若后续多人并发较高，可迁移到 PostgreSQL。
5. 文件存储目录：上传文件、解析文本、OCR 结果、导出文件统一保存在宿主机挂载目录。

建议 Docker 挂载目录：

1. `/data/db`：数据库文件。
2. `/data/uploads`：用户上传的原始文件。
3. `/data/parsed`：解析后的文本、页码、表格、OCR 结果。
4. `/data/exports`：导出的 Excel 和报告。
5. `/data/logs`：系统日志。

NAS 部署时，以上目录映射到 NAS 的持久化共享目录，方便备份和迁移。

## 4. 用户角色

一期按内部工具设计，不做复杂权限。

默认角色：

1. 普通用户：新建项目、上传文件、查看提取结果、人工复核、导出。
2. 管理员：管理基础配置、风险关键词、特殊符号规则、项目删除。

如果一期需要进一步简化，可以先不做账号体系，只保留本地访问和基础操作。

## 8. 文档解析流程

### 8.1 文件接收

系统接收用户上传文件或公告链接，记录文件名、类型、大小、上传时间、所属项目和处理状态。

### 8.2 文本解析

不同文件采用不同解析方式：

1. PDF：优先提取文本层。若文本层不足，再进入 OCR。
2. DOCX：提取段落、表格、页眉页脚。
3. 图片或扫描件：通过 OCR 提取文本。
4. 网页：提取标题、发布时间、正文、附件链接。
5. 压缩包：解压后逐个文件解析。

解析结果应保留来源定位：

1. 文件名。
2. 页码。
3. 章节标题。
4. 表格位置。
5. 原文片段。

### 8.3 结构化抽取

字段抽取采用规则和 AI 结合：

1. 时间、金额、电话、邮箱等格式明确的信息优先使用规则识别。
2. 招标人、代理机构、项目名称等上下文相关字段使用规则加 AI 校验。
3. 长条款、废标条款、星号条款使用规则召回，再由 AI 分类和摘要。
4. 所有抽取结果都必须保留原文依据。

## 9. 废标风险识别逻辑

### 9.1 风险来源

系统需要识别以下来源的风险：

1. 明确包含"废标"的条款。
2. 明确包含"否决投标"的条款。
3. 明确包含"无效投标"或"投标无效"的条款。
4. 符合性审查表中不通过即否决的条款。
5. 资格审查表中的硬性要求。
6. 投标人须知前附表中的实质性要求。
7. 带 `★`、`*`、`▲` 等特殊符号的条款。
8. 文件中说明为实质性要求、不得负偏离、必须满足的条款。
9. 签字盖章、密封、递交、份数、电子签章要求。
10. 保证金缴纳金额、方式、到账时间要求。
11. 报价不得超过最高限价或分项限价要求。
12. 服务期限、交付期、运维响应时间、验收要求。
13. 源代码、知识产权、数据归属、保密要求。
14. 等保、密评、信创、国产化要求。
15. 人员配置、项目负责人、业绩证明要求。

### 9.2 特殊标记条款

系统不能直接假设所有 `★`、`*`、`▲` 条款都会导致废标。处理顺序如下：

1. 先扫描全文，查找特殊符号说明。
2. 判断文件是否说明该符号代表实质性条款、重要参数、不得负偏离或不满足将导致无效投标。
3. 再扫描全文提取带有相同符号的条款。
4. 如果符号说明明确关联废标或无效投标，则标记为高风险。
5. 如果符号说明只是重要参数或评分项，则标记为中风险。
6. 如果未找到符号说明，则标记为需人工确认。

需要识别的符号包括但不限于：

1. `★`
2. `*`
3. `▲`
4. `#`
5. `※`

常见符号说明示例：

1. "带 ★ 号条款为实质性条款，不满足作无效投标处理"。
2. "标注 * 的参数为重要参数"。
3. "带 ▲ 的技术指标不得负偏离"。
4. "不满足星号参数将导致投标无效"。

### 9.3 风险分类

一期内置风险类别：

1. 明确废标/否决投标。
2. 资格审查不通过。
3. 符合性审查不通过。
4. 特殊标记实质性条款。
5. 不允许负偏离。
6. 签字盖章要求。
7. 密封和递交要求。
8. 投标保证金要求。
9. 报价和最高限价要求。
10. 服务期限或交付期要求。
11. 运维响应要求。
12. 源代码、知识产权、数据归属要求。
13. 等保、密评、信创、国产化要求。
14. 人员配置或项目负责人要求。
15. 业绩证明或案例要求。
16. 其他需关注条款。

### 9.4 严重程度

严重程度分为：

1. 高：原文明确说明不满足会废标、否决投标、投标无效，或特殊符号说明明确关联无效投标。
2. 中：原文说明必须满足、不得负偏离、实质性响应，但未直接说明废标后果。
3. 低：需要关注但不一定导致废标，例如评分重要项、服务承诺、资料完整性提醒。
4. 待确认：符号含义或条款后果不明确，需要人工确认。

## 10. 人工复核

人工复核是系统的一期核心能力。

字段和风险均需要支持以下状态：

1. 待复核。
2. 已确认。
3. 已修改。
4. 已忽略。

人工修改后，系统保存：

1. 机器提取值。
2. 人工确认值。
3. 修改时间。
4. 修改人。
5. 备注。

一期如果不做账号体系，修改人可以默认为本地用户或留空。

## 11. 提醒与状态

一期提醒以系统内提醒和颜色预警为主。

时间节点包括：

1. 报名截止。
2. 文件获取截止。
3. 答疑或澄清截止。
4. 保证金缴纳截止。
5. 投标截止。
6. 开标时间。

预警规则：

1. 已过期：红色。
2. 24 小时内截止：红色。
3. 3 天内截止：橙色。
4. 7 天内截止：黄色。
5. 7 天以上：正常。

## 14. 错误处理

常见错误和处理方式：

1. 文件无法解析：标记解析失败，保留原文件，允许用户重试。
2. OCR 失败：提示用户人工查看原文件或重新上传。
3. 字段抽取失败：保留为空，状态为待复核。
4. 多个候选值冲突：展示所有候选值，要求人工确认。
5. 网页抓取失败：允许用户粘贴公告正文或上传网页保存文件。
6. 导出失败：记录错误日志，允许重新导出。

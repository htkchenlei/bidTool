# 模拟评分模块完整开发计划

## 一、Summary（概要）

将现有"一体化 LLM 评分"的模拟评分模块，升级为 9 个子模块协作的分阶段评分体系。核心改造：拆分价格分（规则引擎）、商务分（LLM 抽取 + 规则去重）、技术分（LLM 主观评分）三条独立评分链路；扩展文件格式支持（PDF/图片）；增加证据引用、置信度、模型版本审计；完善同分处理与分阶段进度展示。严格遵守"结果不持久化"硬约束，审计信息随评分结果返回前端并内嵌 docx 报告。

## 二、Current State Analysis（现状分析）

  - 已支持 PDF（pdfplumber，含表格）、DOCX（python-docx，含表格）、图片（pytesseract OCR）、TEXT、HTML

  - `get_all_enabled_model_configs`：获取可用模型列表

### 缺失/待改进
1. 投标文件仅支持 .docx，不支持 PDF/图片（前端硬限制 + 后端 parse_file 调用时硬编码 "DOCX"）
2. 评分项未分类（价格/商务/技术），价格分完全依赖 LLM 判断，无公式计算
3. 无独立投标信息抽取（报价、资质、方案章节未结构化）
4. 商务证据无 LLM 抽取 + 去重计分
5. 技术分一次性整份评分，非按 rubric 子项独立评分
6. 无证据引用（evidence）、置信度（confidence）、模型版本审计
7. 同分未处理（报价低者优先）
8. 进度展示粗糙（仅单一 loading，无分阶段进度）
9. 流式输出中 LLM 响应内容直接 yield 给前端（前端未消费 type=stream 的 chunk，仅 console.log status）

## 三、Assumptions & Decisions（假设与决策）

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 文件格式 | 扩展支持 PDF + 图片 | 用户模块1明确要求"接收 PDF / DOCX / 图片"，doc_parser.py 已有能力 |
| 评分架构 | 分阶段模块化（混合方案） | 用户模块4要求"按评分办法计算价格分"（规则引擎），模块5要求"LLM 抽取+规则去重"，模块6要求"按 rubric 独立评分" |
| 审计日志 | 内存审计 + 报告内嵌 | 项目硬约束"结果不得持久化"，审计信息随评分结果返回，写入 docx |
| 同分处理 | 报价低者优先 | 用户模块7明确要求 |
| 价格分计算方式 | 支持最低价法 + 基准价法 + 线性插值 | 覆盖政府采购常见评分办法 |
| 小微企业扣除 | 默认 6%（可配置） | 政府采购常见扣除比例 |
| LLM 调用 | 价格分纯规则计算；商务证据 LLM 抽取 + 规则去重计分；技术分 LLM 按子项评分 | 平衡精度与性能，避免 LLM 调用过多导致超时 |
| 现有查重功能 | 保留 | `_detect_plagiarism` 已实现，继续使用 |
| 现有 docx 报告 | 增强扩展 | 在现有 `_build_score_report_docx` 基础上增加审计章节 |

#### A10. 重构 `_generate_bid_score_stream` 主流程
新主流程：
```
1. yield status(parsing, 0.05) — 解析投标文件
2. yield status(criteria, 0.15) — 提取评分标准（流式）
3. 对每份投标文件：
   - yield status(extracting, 0.15 + 0.15*(idx/n)) — LLM 抽取投标信息
4. yield status(price, 0.5) — 规则计算价格分
5. yield status(commercial, 0.65) — LLM 抽取商务证据 + 规则计分
6. 对每份投标文件：
   - yield status(technical, 0.65 + 0.25*(idx/n)) — LLM 技术分（流式）
7. yield status(ranking, 0.95) — 汇总排名
8. yield complete — 返回完整结果（含 audit）
```

## 五、Verification Steps（验证步骤）

1. **文件格式**：分别上传 .docx / .pdf / .png 投标文件，确认均能解析并评分
2. **价格分**：上传含不同报价的投标文件，验证价格分按公式计算（非 LLM 判断），小微企业扣除生效
3. **商务证据**：上传含业绩/证书的投标文件，验证 LLM 抽取证据并去重计分
4. **技术分**：验证技术分按子项独立评分，每项有 evidence 和 confidence
5. **同分处理**：构造两个总分相同的投标人，验证报价低者排名靠前
6. **审计信息**：确认结果看板和 docx 报告均含模型版本、置信度、证据引用
7. **进度展示**：确认评分过程中分阶段进度条正确更新
8. **模型切换**：模拟 LLM 调用失败，验证模型切换对话框弹出，切换后继续评分
9. **不持久化**：刷新页面后评分结果清空
10. **UI 一致性**：确认新增 UI 元素使用现有设计系统样式类
11. **现有查重**：验证技术查重功能仍正常工作

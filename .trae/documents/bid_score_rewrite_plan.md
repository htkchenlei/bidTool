# 模拟评分功能重写计划

## 一、需求分析

### 1.1 核心需求

| 需求点 | 描述 | 来源 |
|--------|------|------|
| markitdown 集成 | 使用 `microsoft/markitdown` 将招标文件转换为 markdown 格式 | 用户明确要求 |
| 评分标准提取 | 从 markdown 格式的招标文件中提取评分标准，以表格形式展示 | 用户明确要求 |
| 单文件评分 | 允许上传一份投标文件进行模拟评分 | 用户明确要求 |
| 图片证据验证 | 大模型 OCR 识别图片内容，验证证件名、有效期等信息 | 用户明确要求 |
| 文本内容匹配 | 无图片证据时，根据文字内容匹配程度打分 | 用户明确要求 |
| 文件拆分方案 | 设计投标文件的拆分/转换策略 | 用户明确要求 |
| 提示词设计 | 设计大模型评分的提示词 | 用户明确要求 |

### 1.2 与现有架构的差异

| 维度 | 现有架构 | 新需求 |
|------|----------|--------|
| 投标文件数量 | 最少 3 份，最多 10 份 | 仅需 1 份 |
| 评分模式 | 多投标人对比评分 | 单投标人独立评分 |
| 文档转换 | pdfplumber/python-docx 纯文本提取 | markitdown 转 markdown |
| 图片处理 | 仅 OCR 提取文字 | 视觉模型直接识别 + OCR 兜底 |
| 证据验证 | 仅文本匹配 | 图片证据验证 + 文本匹配双轨 |

## 三、文件拆分方案设计

### 3.1 投标文件结构分析

投标文件通常包含以下章节结构：

```
1. 投标函
2. 商务部分
   ├── 公司资质证书
   ├── 营业执照
   ├── 体系认证证书
   ├── 财务状况证明
   ├── 社保证明
   └── 业绩案例
3. 技术部分
   ├── 技术方案
   ├── 实施方案
   ├── 服务方案
   └── 人员配置
4. 报价部分
   ├── 开标一览表
   └── 分项报价表
```

### 3.2 拆分策略

```python
# 拆分流程：
# 1. 使用 markitdown 将文件转换为 markdown
# 2. 按 markdown 标题层级拆分章节
# 3. 提取每个章节中的图片（保留章节上下文）
# 4. 构建章节-图片映射表

章节结构 = {
    "商务部分": {
        "子章节": ["公司资质", "业绩案例", "社保证明", "财务状况"],
        "图片类型": ["cert", "case", "social_security", "finance"]
    },
    "技术部分": {
        "子章节": ["技术方案", "实施方案", "服务方案"],
        "图片类型": []  # 技术部分主要为文字描述
    },
    "报价部分": {
        "子章节": ["开标一览表", "分项报价"],
        "图片类型": ["price_table"]
    }
}
```

### 3.3 图片提取方案

| 文件类型 | 图片提取方式 | 技术实现 |
|----------|-------------|----------|
| DOCX | 提取内联图片和嵌入式对象 | python-docx 的 `inline_shapes` / `part.rels` |
| PDF | 每页转换为图片 + 提取嵌入式图片 | pdfplumber 或 PyPDF2 |
| 图片文件 | 直接使用 | 无需转换 |

## 四、评分引擎设计

### 4.1 双轨评分模型

```
评分项
  │
  ├─ 需要图片证据？
  │     ├─ 是 ──► 图片OCR识别 ──► 验证证件名/有效期 ──► 打分
  │     └─ 否 ──► 文本内容匹配 ──► 匹配程度评估 ──► 打分
  │
  └─ 评分结果: { score, max_score, evidence_type, matched_content, confidence }
```

### 4.2 图片证据验证流程

```
图片文件(.png/.jpg)
        │
        ▼
┌─────────────────────────────┐
│ 检查模型是否支持视觉能力     │
│ supports_vision == true?    │
└─────────────────────────────┘
        │
    ┌───┴───┐
    │       │
   是       否
    │       │
    ▼       ▼
┌─────────┐ ┌─────────────────┐
│ 视觉模型 │ │ pytesseract OCR │
│ 直接识别 │ │ 提取文字后识别   │
└─────────┘ └─────────────────┘
        │       │
        └───┬───┘
            ▼
┌─────────────────────────────┐
│ LLM 验证: 证件名/有效期/颁发机构 │
│ 返回: { name, type, issuer, expire, valid } │
└─────────────────────────────┘
```

### 4.3 文本内容匹配流程

```
评分项要求 + 投标文件章节内容
        │
        ▼
┌─────────────────────────────┐
│ LLM 内容匹配评分            │
│ - 关键词匹配度              │
│ - 内容完整性                │
│ - 要求满足程度              │
└─────────────────────────────┘
        │
        ▼
评分结果: { score, reason, evidence, confidence }
```

## 五、提示词设计

### 5.1 评分标准提取提示词

```
系统提示词:
你是一位专业的政府采购评审专家。请从提供的招标文件内容中提取评分标准的各项要求。
只返回 JSON 对象本身，不要包含 ``` 代码块或任何解释文字。

输出格式:
{
  "criteria": [
    {
      "id": "唯一标识",
      "name": "评分项名称",
      "max_score": 满分,
      "category": "price | commercial | technical",
      "requires_image": true/false,
      "description": "评分要求描述",
      "sub_items": [
        {"name": "子项名", "max_score": 分值, "requires_image": true/false}
      ],
      "keywords": ["关键词1", "关键词2"]
    }
  ],
  "total_max": 总满分,
  "price_rule": {...}
}

要求:
1. requires_image: 是否需要图片证据（如资质证书、社保证明等需要图片验证的项为true）
2. keywords: 用于在投标文件中定位相关内容的关键词
3. 尽可能完整地提取所有评分项
```

### 5.2 图片证据验证提示词

```
系统提示词:
你是一位专业的资质证书验证专家。请识别图片中的证书内容并验证其有效性。
只返回 JSON 对象本身。

输出格式:
{
  "cert_name": "证书全称",
  "cert_type": "证书类型",
  "issuer": "颁发机构",
  "expire_date": "到期日期(YYYY-MM-DD)",
  "is_valid": true/false,
  "confidence": 0.0-1.0,
  "extracted_text": "图片中提取的关键文字"
}

要求:
1. 如果未明确标注有效期，则认为在有效期内（is_valid=true）
2. 日期格式统一为 YYYY-MM-DD
3. confidence 表示识别的置信度
```

### 5.3 文本内容匹配评分提示词

```
系统提示词:
你是一位专业的评标专家。请根据评分标准，对照投标文件内容给出评分。
只返回 JSON 对象。

输出格式:
{
  "score": 得分,
  "max_score": 满分,
  "reason": "打分依据",
  "evidence": "投标文件中的具体引用",
  "confidence": 0.0-1.0,
  "match_details": [
    {"keyword": "关键词", "found": true/false, "context": "匹配上下文"}
  ]
}

要求:
1. 根据内容匹配程度、完整性、准确性综合打分
2. evidence 必须引用投标文件中的具体原文
3. match_details 列出每个关键词的匹配情况
```

## 九、风险与应对

| 风险 | 等级 | 应对措施 |
|------|------|----------|
| markitdown 对某些文档格式支持不佳 | 中 | 保留 doc_parser.py 作为兜底方案 |
| 视觉模型 API 调用失败 | 高 | 实现 OCR 兜底 + 模型切换机制 |
| 投标文件过大导致内存溢出 | 中 | 分片处理 + 流式传输 |
| LLM 返回格式不稳定 | 中 | 增强 JSON 解析容错 + 重试机制 |
| 图片提取不完整 | 低 | 支持多种提取方式 + 人工补充 |

## 十一、代码规范

### 命名规范

- 模块名：小写加下划线（markdown_converter.py）
- 函数名：小写加下划线（convert_to_markdown）
- 类名：大驼峰（BidParser）
- 变量名：小写加下划线（bid_sections）

### 错误处理

- 所有 LLM 调用必须有 try-catch
- 返回统一的错误格式
- 记录详细的错误日志

### 性能优化

- 使用流式输出减少等待时间
- 大文件分片处理
- 图片压缩后再传输

1. ✅ 招标文件成功转换为 markdown 格式

3. ✅ 单份投标文件可完成模拟评分

5. ✅ 文本内容匹配程度正确打分
6. ✅ 结果不持久化，刷新页面后清空
7. ✅ 模型调用失败时可切换模型继续评分
